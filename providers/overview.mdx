FreeLLMAPI aggregates free inference tiers from 14 AI providers behind a single OpenAI-compatible endpoint. Each provider contributes a slice of capacity — from a few million tokens per month to tens of millions — and the router combines them into a single pool of roughly **1.3 billion tokens per month**. You add API keys for whichever providers you want, and the router picks the best available model for every request, falling over to the next provider when one is rate-limited or unavailable.

<Note>
  Free tier limits change without notice. Providers regularly tighten, loosen, or remove free quotas. The numbers below reflect the live catalog as of **April 2026** after live-probing every model against real free-tier keys. If you start seeing unexpected 429s or auth errors, check the provider's current docs and re-seed the catalog using the scripts in `server/src/scripts/`.
</Note>

## The `auto` model

When you set `model="auto"` in a request, the router selects the highest-priority model that currently has a healthy key and is under all its rate limits. Priority is determined by the **fallback chain**, which you can reorder from the admin dashboard. By default the chain is ordered by `intelligence_rank` ascending, so the strongest available model is always tried first.

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:3001/v1",
    api_key="freellmapi-your-unified-key",
)

# Let the router pick — uses the top available model right now
resp = client.chat.completions.create(
    model="auto",
    messages=[{"role": "user", "content": "Explain backpressure in 2 sentences."}],
)

# Or target a specific model by its model ID
resp = client.chat.completions.create(
    model="gemini-2.5-flash",
    messages=[{"role": "user", "content": "Explain backpressure in 2 sentences."}],
)
```

Every response includes an `X-Routed-Via: <platform>/<model>` header so you can see which provider actually served the call.

## Getting API keys

Add keys from each provider's console. The FreeLLMAPI dashboard (**Keys** page) accepts multiple keys per provider and round-robins across them.

<Note>
  `zhipu`, `moonshot`, and `minimax` are supported as routing targets but cannot be added via `POST /api/keys` — their platform names are not in the accepted list for the key management endpoint. Add them directly through the dashboard instead.
</Note>
